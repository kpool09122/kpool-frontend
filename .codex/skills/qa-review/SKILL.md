# QA Review

## 目的

kpool-frontend の変更を、実際の利用者が触る画面・操作・表示品質の観点でレビューする。実装の良し悪しよりも、ユーザーに見える破綻、操作不能、誤表示、状態遷移の不整合を優先して指摘する。

## 進め方

1. 変更されたページ、コンポーネント、ゲートウェイ、E2Eテストを確認し、影響するユーザーフローを特定する。
2. 既存の画面構成を `src/app`, `src/components/Wiki`, `src/i18n`, `tests/e2e` から確認する。
3. 差分が次のどのフローに影響するかを明示してからレビューする。
   - ホームのWiki一覧、検索、フィルタ、ページング
   - Wiki詳細、言語付きURL、旧slug互換URL、空/エラー状態
   - Wiki編集、セクション/ブロック編集、画像ライブラリ、関連プロフィール
   - ログイン、SSOリダイレクト、メールログイン、登録
   - mypage のPrincipal作成、下書き、レビュー、公開、翻訳、画像レビュー
   - ロケール解決、ヘッダー、テーマ切替、モバイルメニュー
4. 指摘は再現条件、ユーザー影響、期待挙動を短く書く。コードの好みだけの指摘は避ける。

## 重点観点

- 操作後の状態が画面に即時反映されるか。React Query のキャッシュ更新、タブ切替、ブラウザバック後の再操作を確認する。
- 権限やPrincipal状態ごとに、表示すべき操作だけが表示されるか。レビュー権限、公開権限、画像ポリシーの有無を混同していないか見る。
- ローディング、空状態、エラー状態、未認証状態がユーザーに理解できる文言で表示されるか。
- `ja`, `en`, `ko` の辞書キー追加漏れ、固定日本語、URL言語と表示言語のズレがないか確認する。
- Wiki編集では、ネストしたセクション、複数ブロック、画像/埋め込み/テーブル/プロフィールカードの操作が壊れていないか見る。
- レスポンシブ表示でヘッダー、フォーム、カード、編集サイドバー、画像ライブラリが重ならないか確認する。
- ボタン、リンク、フォーム、ダイアログがアクセシブルな名前を持ち、キーボード操作やフォーカス復帰が自然かを見る。

## 検証候補

- 既存E2Eに近い変更なら `pnpm test:e2e`、狭い画面挙動なら対象specを優先する。
- コンポーネント単体の表示分岐なら該当 `*.test.tsx` と Storybook story の更新要否を見る。
- 見た目や操作性の不安がある場合は、ローカル起動後にブラウザで該当フローを確認する。

## 出力

レビュー結果は重大度順に並べる。各指摘にはファイル/行、問題、ユーザー影響、確認したい挙動または修正方向を含める。問題が見つからない場合も、確認した範囲と残るQAリスクを短く書く。
