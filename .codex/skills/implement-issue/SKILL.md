# GitHub Issue 実装スキル

あなたは GitHub Issue を元に設計・計画・実装・検証を進める実装担当です。
Issue番号を受け取り、以下のフェーズを順番に実行します。Issue番号が不明なら先に確認します。

## プロジェクト情報

- リポジトリ: `kpool09122/kpool-frontend`
- 言語/FW: TypeScript / Next.js / React
- パッケージマネージャ: `pnpm`
- 主な確認コマンド: `task check`, `pnpm build`, `task test:unit`, `task test:e2e`
- 主なパス:
  - `src/app/`
  - `src/types/`
  - `src/test/`
  - `tests/e2e/`

## フェーズ1: Issue取得と理解

### Step 1: Issue情報の取得

`gh issue view <Issue番号> --repo kpool09122/kpool-frontend --json title,body,labels,assignees`

### Step 2: Issue内容の確認

以下を簡潔に整理します。

- タイトル
- 概要
- 対象領域
- 実装ステップ
- 受け入れ条件
- 不足している前提や曖昧点

Issueが十分具体的なら、そのまま実装に進みます。受け入れ条件や対象範囲が曖昧なら、不足分だけ確認します。

### Step 3: 作業ブランチの作成

必要なら `{Issue番号}-{issue-summary}` 形式のブランチを作ります。

## フェーズ2: 実装計画の作成

実装前に、最低限以下を整理します。

1. 関連ファイルと既存構造
2. 参考にすべき既存実装
3. 影響範囲
4. テスト追加先
5. UI とロジックの責務分割
6. API Response の型定義の参照元

そのうえで、以下を含む具体的な実装計画を作ります。

- 作成/変更するファイル
- 実装順序
- コンポーネント、純粋関数、カスタム hooks の責務分担
- `src/types/` 配下の zod 定義の利用方針
- ユニットテストと E2E テストの追加方針
- 検証方針

計画時は特に以下を確認します。

- UI 表示ロジックを純粋関数へ切り出せるか
- 副作用が必要な場合にカスタム hooks へ閉じ込められるか
- `useEffect` を使わずに成立する構成か
- ロジック部分を TDD で進められるか
- E2E テストをどこに追加するか
- API Response の型定義を `src/types/*.ts` の zod スキーマから参照できるか

Issueが十分に具体的なら、計画作成後は追加確認なしで実装に進みます。

## フェーズ3: 実装

### 選択肢A: このセッションで直接実装

変更が小さい、またはクリティカルパス上の作業なら、そのままローカルで実装します。

### 選択肢B: サブエージェントに委譲

サブエージェントが利用可能で、作業を分離する価値がある場合は `spawn_agent` を使って実装を委譲します。

委譲時は以下を明示します。

- Issue番号とタイトル
- 実装計画の全文
- 参照すべき既存ファイル
- 変更責任を持つ範囲
- UI とロジック分離の方針
- TDD と E2E 必須の方針

サブエージェントが使えない、または分離の利点が薄い場合は自分で実装します。

### ロジック実装の進め方

ロジック部分は TDD で実装します。

1. まず失敗するテストを書く
2. 最小実装で通す
3. リファクタリングして責務分離を整える

純粋関数で切り出せるロジックは、コンポーネントから分離したうえでテストを書きます。

### 型定義の扱い

API Response の型定義は新規に場当たりで書かず、まず `src/types/` 配下の zod 定義を確認します。
レスポンス形状を扱う場合は、その zod スキーマやそこから導出される型を利用することを優先します。

## フェーズ4: 統合と検証

実装後、以下を確認します。

- `task check` が通る
- `pnpm build` が通る
- `task test:unit` が通る
- `task test:e2e` が通る
- 変更内容が Issue の受け入れ条件に合っている

E2E テストは必須です。UI変更の有無にかかわらず、Issueの変更内容を担保できる E2E シナリオ追加または更新を検討し、不要と判断する場合は理由を明確に説明します。

UI変更がある場合は、手動確認観点も整理して報告します。

サブエージェントを使った場合は、返却された差分・結果を確認してから統合します。

## フェーズ5: 結果報告

以下を報告します。

- 実装内容のサマリー
- 実行した検証
- 追加したユニットテストと E2E テスト
- 未実施の検証があればその理由
- 次のステップ

## フロントエンド実装原則

- UI とロジックは分離する。JSX内で複雑な分岐や計算を増やしすぎない
- まず純粋関数で記述できるかを検討し、可能なら関数として切り出す
- ロジック部分は TDD で実装する
- 副作用がある処理は、コンポーネント本体に直接書かずカスタム hooks に閉じ込める
- `useEffect` は使わない。別の設計で解決する
- 関数型のスタイルを優先し、データ変換・条件分岐・整形を小さな関数で表現する
- クラスは原則使わない。ただし責務が明確で、本当に必要なら例外的に許容する
- API Response の型は `src/types/` 配下の zod 定義を利用する
- E2E テストは必須とし、変更がユーザー体験に与える影響を必ず検証する
- 既存の App Router / React / 型定義 / テストパターンを壊さない

## 重要な原則

- Issueの実装指示を最大限尊重する
- 既存パターンを踏襲しつつ、責務分離を改善する
- 実装前に純粋関数化できる部分を必ず探す
- `useEffect` を前提に設計しない
- ロジックはテスト先行で進める
- API Response の型は既存 zod 定義を優先して再利用する
- E2E テストを実装の完了条件に含める
- 計画は具体的にする
- サブエージェントは手段であり、不要なら使わない
